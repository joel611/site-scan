## Context

The report is a single self-contained HTML file generated by `src/html-report.ts`. It embeds `GRAPH_DATA` (nodes + edges) inline and uses D3 for the force-directed graph. Currently node click calls `window.open(d.url, '_blank')` — there is no detail view. All data needed (node metadata, edge source/target, dead status) already lives in `GRAPH_DATA`; no backend or new data pipeline is required.

## Goals / Non-Goals

**Goals:**
- Replace node click behavior with a detail panel showing sources (outbound) and backlinks (inbound)
- Surface dead link origins: if this node links to a dead node, mark that target as dead in the Sources list
- Allow panel-to-graph navigation: clicking a link entry in the panel selects/centers that node
- Provide "Visit page" button in the panel as the replacement for the old click-to-open behavior
- Dismiss panel with Escape key or backdrop click

**Non-Goals:**
- No changes to crawler, graph-builder, or types — graph data model is unchanged
- No persistent panel state across page loads
- No deep-link routing to a specific node on load

## Decisions

### 1. Panel as overlay, not side-split

**Decision**: Render the panel as a fixed overlay anchored to the right side, layered above the graph canvas.

**Rationale**: Resizing the graph triggers a D3 simulation re-layout, which is jarring. An overlay avoids any layout reflow while still leaving most of the graph visible.

**Alternative**: Push-layout (panel shrinks graph container). Rejected — forces simulation restart and users lose spatial context of selected node.

### 2. Build adjacency data at runtime from `GRAPH_DATA.edges`

**Decision**: When a node is clicked, compute its sources and backlinks by filtering `GRAPH_DATA.edges` where `edge.source === node.id` (sources) or `edge.target === node.id` (backlinks). Resolve node metadata by looking up `GRAPH_DATA.nodes`.

**Rationale**: `GRAPH_DATA` already carries all required information. Precomputing a lookup map on init is O(E) and negligible.

**Alternative**: Pre-index adjacency at scan time and embed in node data. Overkill — this is a client-side-only HTML report.

### 3. Panel-to-graph navigation via simulation positioning

**Decision**: Clicking a node entry in the panel calls a `focusNode(id)` function that: (a) sets that node as `selected` state, (b) transitions the SVG zoom transform to center on it, (c) updates node stroke to highlight selected state.

**Rationale**: D3 zoom has a programmatic `.transform()` API. Centering + highlighting makes it obvious which node corresponds to the panel entry.

## Risks / Trade-offs

- **Large edge lists**: Pages with hundreds of inbound links will produce long panel lists → Mitigation: cap display at 50 entries with a "+N more" indicator
- **Panel covers graph nodes**: The overlay will occlude the right side of the graph → Mitigation: panel width fixed at 320px; users can drag nodes out from under it
- **Escape key conflicts**: Other listeners could intercept Escape → low risk in a self-contained HTML report with no external JS frameworks

## Migration Plan

Self-contained HTML change. No migration needed — existing scans remain valid. Re-run the scanner to generate a report with the new panel.
